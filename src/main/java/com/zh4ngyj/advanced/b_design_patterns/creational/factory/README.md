# 设计模式: 工厂模式 (Factory)

工厂模式是一种创建型模式，它提供了一种创建对象的最佳方式，允许子类决定实例化哪一个类，将对象的创建与使用分离。

## 1. 简单工厂模式 (Simple Factory)
- **定义**: 一个工厂类根据传入的参数，动态决定应该创建哪一种产品类的实例。
- **角色**:
    - **工厂 (Factory)**: 负责创建具体产品的核心类。
    - **抽象产品 (Abstract Product)**: 定义产品的公共接口。
    - **具体产品 (Concrete Product)**: 实现抽象产品接口。
- **缺点**: 不符合开闭原则。每增加一个新产品，都需要修改工厂类的代码。

## 2. 工厂方法模式 (Factory Method)
- **定义**: 定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。
- **角色**:
    - **抽象工厂 (Abstract Factory)**: 声明创建产品的工厂方法。
    - **具体工厂 (Concrete Factory)**: 实现工厂方法，创建具体产品。
    - **抽象产品** 和 **具体产品**。
- **优点**: 符合开闭原则。增加新产品时，只需增加一个具体产品类和一个对应的具体工厂类，无需修改现有代码。

## 3. 抽象工厂模式 (Abstract Factory)
- **定义**: 提供一个接口，用于创建**一系列相关或相互依赖的对象**（产品族），而无需指定它们具体的类。
- **与工厂方法的区别**:
    - **工厂方法**: 一个工厂只生产一种产品。
    - **抽象工厂**: 一个工厂生产一个产品族（多个不同种类的产品）。
- **角色**:
    - **抽象工厂**: 定义创建多个产品（一个产品族）的接口。
    - **具体工厂**: 实现抽象工厂，创建具体的产品族。
    - **多组抽象产品/具体产品**。
- **优点**: 易于交换产品族。
- **缺点**: 难以增加新的产品种类。如果产品族需要增加一个新的产品，所有工厂类都需要修改。
