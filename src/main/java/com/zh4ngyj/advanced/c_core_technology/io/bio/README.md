# I/O 模型: BIO (Blocking I/O)

BIO (Blocking I/O)，即同步阻塞 I/O，是 Java 中最传统、最简单的 I/O 模型。

## 核心模型
- **一对一连接**: 服务器端为每个客户端连接都创建一个新的线程进行处理。
- **阻塞特性**:
    - 当一个连接建立后，如果客户端没有发送数据，那么服务器端负责处理该连接的线程将在 `read()` 方法处被**阻塞**，直到有数据可读。
    - 在数据处理期间，该线程也无法接受其他客户端的连接。

## 工作流程 (服务器端)
1.  服务器启动一个 `ServerSocket`，并调用 `accept()` 方法，阻塞地等待客户端连接。
2.  当一个客户端连接成功时，`accept()` 方法返回一个 `Socket` 对象。
3.  服务器为这个 `Socket` 创建一个新的线程。
4.  该线程通过 `Socket` 的 `getInputStream()` 读取数据，这个 `read()` 操作是阻塞的。
5.  处理完数据后，通过 `getOutputStream()` 向客户端写回数据。
6.  服务器主线程继续调用 `accept()` 等待下一个连接。

## 优缺点

### 优点
- **模型简单**: 编程模型直观，易于理解和实现。

### 缺点
- **性能瓶颈**:
    - **线程开销大**: 每个连接都需要一个独立的线程，当并发连接数很高时，系统需要创建大量线程，导致资源消耗巨大，上下文切换频繁。
    - **阻塞导致资源浪费**: 如果大量连接处于不活跃状态（长连接但没有数据交互），对应的线程会被阻塞，造成严重的资源浪费。
- **扩展性差**: 受限于线程数量，无法支持高并发场景（C10K 问题）。

BIO 模型适用于连接数较少且固定的架构，不适合需要处理大量并发连接的现代互联网应用。
